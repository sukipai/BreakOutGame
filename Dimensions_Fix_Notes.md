# 关于窗口尺寸与物理像素（Framebuffer）的修复总结

在开发具有“后期处理”功能的 OpenGL 程序时，经常会遇到“显示内容放大/缩小”或“黑屏”的问题，特别是在 Retina 屏幕（高 DPI）的设备上。本次修复主要解决了以下核心问题：

## 1. 核心概念：逻辑坐标 vs 物理像素

*   **逻辑坐标 (Logic Size/Window Size)**：
    *   这是你在代码中定义的坐标系，例如 `800x600`。
    *   你的游戏逻辑（如挡板宽度为 100、球半径为 12.5）和 **正交投影矩阵 (Ortho Projection)** 都是基于这个坐标系的。
*   **物理像素 (Framebuffer Size)**：
    *   这是屏幕实际的像素点。在 Retina 屏幕上，物理像素通常是窗口尺寸的 2 倍（例如 `1600x1200`）。
    *   **OpenGL 的 Viewport 和 Framebuffer (FBO) 必须基于物理像素设置。**

## 2. 之前出现的问题及原因

### 问题 A：内容被放大且错位
*   **原因**：后期处理的 FBO 尺寸设置为了逻辑尺寸（800x600），但渲染时的视口（Viewport）却是物理像素尺寸（1600x1200）。
*   **后果**：OpenGL 尝试在一个 1600x1200 的视口中渲染，但目标 FBO 只有 800x600，导致只有左下角的 1/4 内容被画进了 FBO，采样出来后看起来就像是被放大了。

### 问题 B：物体变小了（你最后提到的现象）
*   **原因**：为了修复分辨率问题，我们将投影矩阵也改成了基于物理像素（1600x1200）。
*   **后果**：由于你的游戏对象尺寸（如 Width=100）是基于 800 坐标系设计的，现在投影矩阵变成了 0 到 1600，100 宽度的物体在视觉上就显得只有原来的一半大。

### 问题 C：黑屏/纹理不显示
*   **原因 1**：`FBO` 变量没有通过 `glGenFramebuffers` 实际生成，导致绑定到了默认的 0 号缓冲。
*   **原因 2**：纹理过滤设为了 `GL_LINEAR_MIPMAP_LINEAR`，但 FBO 纹理并没有生成 Mipmap，导致采样失败。

## 3. 修改方案总结

### 尺寸管理分离
我们在 `Game` 类中显式区分了这两个尺寸：
*   `Width`, `Height`：始终保持 **800x600**。用于游戏逻辑和投影矩阵。
*   `FBWidth`, `FBHeight`：通过 `glfwGetFramebufferSize` 获取的 **实际像素值**。用于 FBO 和 Viewport。

### 代码改动点
1.  **PostProcessor 初始化**：现在使用 `FBWidth/FBHeight` 创建，确保 FBO 分辨率足够高。
2.  **Projection 矩阵**：恢复使用 `Width/Height`，确保游戏物体的大小比例正确。
3.  **Viewport 管理**：
    *   在 `PostProcessor::BeginRender` 中：`glViewport(0, 0, FBWidth, FBHeight)`，确保渲染到 FBO 时的范围正确。
    *   在 `PostProcessor::Render` 中：同样强制设置一次视口，确保将 FBO 渲染回屏幕时的范围也正确。
4.  **纹理参数优化**：将 FBO 纹理的过滤方式改回 `GL_LINEAR`，因为后期处理通常不需要（也不生成）Mipmap。

## 4. 学习建议
在以后处理屏幕尺寸时，请记住这个原则：
> **游戏逻辑（哪里是 0.0，哪里是 800.0）用逻辑坐标；硬件渲染（Viewport 范围，FBO 大小）用物理像素。**

你可以通过 `glfwGetFramebufferSize` 永远获取最准确的物理像素大小。
